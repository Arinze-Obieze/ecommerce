Pagination, Caching, and Data Access Implementation Plan
Date: 2026-04-20

Current status
--------------
The project already has a good rendering foundation:

- app/(storefront)/layout.js uses revalidate = 300.
- app/(content)/layout.js is static.
- app/(account), app/(admin), and app/(store-console) are dynamic and no-store.
- API cache response headers are centralized in utils/platform/api-response.js.
- Redis support exists in utils/platform/redis.js.
- Rate limiting exists in utils/platform/rate-limit.js.

The main remaining gap is consistency. Some pages and route handlers query Supabase directly, some use feature clients, and public/private caching policy is not yet centralized enough for quick fixes.

Recommended target architecture
-------------------------------
Create one consistent data path:

1. Route handlers parse request input and return JSON.
2. Server pages call feature server modules for server-side data.
3. Client components call feature API clients.
4. Feature server modules own Supabase queries and cache behavior.
5. Shared cache policy and tags live under utils/platform.

Recommended folders:

- utils/platform/cache-policy.js
- utils/platform/cache-tags.js
- utils/platform/cached-fetch.js
- features/catalog/server/
- features/storefront/stores/server/
- features/account/server/
- features/store-console/server/
- features/admin/server/


Central cache policy
--------------------
Add a cache policy module at:

utils/platform/cache-policy.js

Suggested policies:

- publicCatalog
  CDN TTL: 120 seconds
  Browser TTL: 0 seconds
  Revalidation: must-revalidate
  Use for product lists, categories, collections, public store lists.

- publicDetail
  CDN TTL: 60 seconds
  Browser TTL: 0 seconds
  Revalidation: must-revalidate
  Use for product detail and public store detail pages.

- publicShort
  CDN TTL: 30 seconds
  Browser TTL: 0 seconds
  Revalidation: must-revalidate
  Use for hero banners, top stores, and homepage merchandising that changes more often.

- privateUser
  no-store
  Use for account, cart, wishlist, addresses, orders, notifications.

- storeConsole
  no-store
  Use for seller dashboard, inventory, products, orders, payouts, team, analytics.

- adminOps
  no-store
  Use for admin dashboards, reviews, sellers, products, logs, escrow, orders.

- mutation
  no-store
  Use for all POST, PATCH, PUT, and DELETE routes.


Central cache tags
------------------
Add a tag registry at:

utils/platform/cache-tags.js

Suggested tags:

- products
- products:list
- products:detail:{id}
- products:store:{storeId}
- categories
- collections
- stores
- stores:list
- stores:top
- stores:detail:{id}
- storefront:home
- return-policy
- reviews:product:{productId}

Mutation routes should invalidate related tags after writes.

Examples:

- Product create/update/delete:
  Invalidate products, products:list, products:store:{storeId}, stores:detail:{storeId}

- Store settings update:
  Invalidate stores, stores:list, stores:top, stores:detail:{storeId}

- Review create/update/moderate:
  Invalidate reviews:product:{productId}, products:detail:{productId}

- Category or collection changes:
  Invalidate categories, collections, products:list, storefront:home


Redis usage
-----------
Redis should be used deliberately, not everywhere.

Good Redis use cases:

- Rate limiting.
- Expensive recommendation ranking.
- Aggregated metrics.
- Hot computed data that is expensive to rebuild.
- Temporary payment/session locks if needed.

Avoid Redis for:

- Simple public lists that Next/CDN caching can handle.
- Private user data.
- Seller/admin dashboards that must always be fresh.


Pagination standard
-------------------
Every list endpoint should return the same shape:

{
  "success": true,
  "data": [],
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 0,
    "totalPages": 1,
    "hasNextPage": false,
    "hasPreviousPage": false
  }
}

Endpoints that should follow this standard:

- /api/products
- /api/stores
- /api/admin/products
- /api/admin/orders
- /api/admin/stores
- /api/admin/logs
- /api/store/products
- /api/store/orders
- /api/store/inventory
- /api/store/payouts
- /api/account/notifications
- /api/account/orders if added later


Streaming and Suspense plan
---------------------------
Use streaming mostly on public storefront pages.

Good candidates:

- Homepage:
  Render shell and hero immediately.
  Stream top stores, recommended products, recently viewed, and merchandising sections.

- Product detail page:
  Render product primary data first.
  Stream reviews, related products, and recently viewed products.

- Store profile page:
  Render store header first.
  Stream store products and reviews separately.

Avoid overusing streaming in:

- Checkout.
- Admin dashboards.
- Seller order pages.
- Inventory operations.

For operational pages, no-store plus explicit loading states is safer and easier to reason about.

Rules of thumb
--------------
- Public catalog/storefront data can be cached briefly.
- Private account data must be no-store.
- Admin and seller console data must be no-store.
- Mutations must be no-store.
- Route handlers should stay thin.
- Business queries should live in feature server modules.
- Cache policy should be changed in one central file, not scattered across pages.
- Pagination shape should be consistent everywhere.
